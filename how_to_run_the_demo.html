The fast path

One-click install

Don't want to type commands? Use the bundled installer.

Windows

Double-click install.bat in this folder.

It opens a console, verifies Python, creates a virtual environment, installs the dependency, and runs a smoke test. ~20 seconds.

To re-run the demo later: double-click run_demo.bat.

Linux / macOS

chmod +x install.sh run_demo.sh
./install.sh

Same flow on POSIX systems. To re-run the demo later: ./run_demo.sh.

No admin / sudo. No registry changes. Everything lives inside this folder. If you'd rather run the steps yourself, the manual walkthrough is on the next slides.

Before you start

Prerequisites

Python 3.10+

Verify with python --version (Windows) or python3 --version (Linux/macOS).

If missing, install from python.org and ensure it's on your PATH.

A terminal

Windows: PowerShell (built-in).

Linux / macOS: bash / zsh (built-in).

~ 5 MB free disk

The harness is tiny — only one Python dependency (jsonschema).

Internet needed once for pip install; offline thereafter.

Step 1 of 5
1

Open the project in a terminal

Extract the package and change into the harness folder.

Windows (PowerShell)

PS> cd path\to\aurora_satcom_demo\satcom_requirements_agent_harness

Linux / macOS

$ cd path/to/aurora_satcom_demo/satcom_requirements_agent_harness

You should see config.json, requirements.txt, input/, prompts/, src/, tests/.

Step 2 of 5
2

Create a virtual environment

Isolates this project's single dependency from your system Python — recommended best practice.

Windows

PS> python -m venv .venv

Linux / macOS

$ python3 -m venv .venv
$ source .venv/bin/activate

A new folder .venv/ is created. Nothing outside this folder is touched.

Step 3 of 5
3

Install dependencies

Only one library is needed: jsonschema, used to validate every stage output.

Windows

PS> .\.venv\Scripts\python.exe -m pip install -r requirements.txt

Linux / macOS (with venv activated)

$ pip install -r requirements.txt

This is the only step that touches the network. After this, the pipeline runs fully offline.

Step 4 of 5
4

Run the pipeline

One command. The runner is src/main.py.

Windows

PS> .\.venv\Scripts\python.exe src\main.py --config config.json

Linux / macOS

$ python src/main.py --config config.json

The pipeline runs all 5 stages, validates outputs, computes the score, and prints score_report.json to your terminal.

Total runtime: well under 1 second.

Step 5 of 5
5

Check the result

You should see this in your terminal:

{
  "category_scores": {
    "traceability_accuracy": 100,
    "deduplication_quality": 100,
    "conflict_detection":    100,
    "feature_quality":       100,
    "pbi_quality":           100,
    "json_validity":         100
  },
  "final_score": 100.0,
  "passed":      true
}

If you see "passed": true with a score ≥ 80 — you're done. ✅

Outputs

Open any of these in a text editor or JSON viewer to inspect the pipeline's reasoning end-to-end.

Highlights

What to look at first

Traceability

Open stage3_sss.json and look at any item — every entry has both source_srd_ids and source_customer_ids. Every system-level requirement traces back to its customer.

Conflict detection

Find SSS-DATA-001 — its conflict_notes capture a real disagreement: Customer A wants 7-year log retention, Customer B wants 24-month billing retention.

Dev-ready PBIs

Open stage5_pbis.json. Every acceptance_criteria entry contains Given / When / Then — ready for sprint planning.

Make it your own

Try modifications

Get a feel for how the pipeline reacts.

  1. Edit a customer requirement in input/customer_requirements/01_Customer_A_Defense_Requirements.md — re-run, watch traceability propagate through all five stages.
  2. Add a fourth customer file (e.g. 04_Customer_D_Maritime_Requirements.md) using the same D-REQ-001 ID format — re-run, watch new SRD and SSS items appear automatically.
  3. Introduce a conflict between two customers (different SLAs, retention windows, throughput numbers) — watch conflict_notes get populated in stage3_sss.json.
  4. Break traceability intentionally — drop a requirement ID — watch the validator catch the dangling reference and the score drop.
Going to production

Swapping in a real local LLM

The demo runs in deterministic mock mode. To use a real on-prem LLM (NVIDIA NIM, vLLM, TensorRT-LLM, Llama / Mistral wrapper):

  1. In config.json, set "mode": "local" and point llm.endpoint at your inference service.
  2. Implement LocalLLMClient.generate_json() in src/llm_client.py — POST prompt + payload to your endpoint, parse JSON, return.
  3. Keep schemas.py, validators.py, scoring.py, and pipeline.py untouched.
  4. Re-run the same command. The runner picks up the new client automatically.

Validation runs downstream of generation, so a misbehaving model is caught before scoring — bad output → low score → CI gate fails.

Troubleshooting

Common issues & fixes

"python is not recognized"

Python isn't on your PATH. Reinstall from python.org and tick "Add Python to PATH".

"pip install" fails behind a corporate proxy

Configure pip with your proxy: pip --proxy=http://proxy:8080 install -r requirements.txt, or use a pre-cached wheel.

PowerShell blocks .venv activation

You don't need to activate on Windows — invoke directly: .\.venv\Scripts\python.exe …

"Permission denied" on Linux/macOS

Make sure you're in the harness folder and the venv is activated. Try chmod +x if needed.

Score below 80

Open validation_report.json — every error tells you exactly which artifact and which field failed.

Want fresh outputs

Delete the outputs/ folder. The next run regenerates all 8 files deterministically.
