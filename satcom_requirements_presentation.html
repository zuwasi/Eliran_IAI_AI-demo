01 • The Problem

From customer wishlists to engineering reality

Three customers. Conflicting demands. Hundreds of pages of requirements docs.

Customer A — Defense

Encryption, signed updates, 7-year audit logs, role-based access.

Customer B — Commercial Telecom

Bandwidth tiers, billing accuracy, SLA reporting, 24-month usage retention.

Customer C — Emergency Response

Rapid deploy, simplified dashboards, priority traffic for first responders.

Manually merging these into a unified system specification takes weeks and is error-prone. The harness does it deterministically in under a second.

02 • Architecture

The 5-stage pipeline

STAGE 1
Ingestion
Regex extracts customer reqs from .md files. Deterministic.
STAGE 2
SRD
Engineering "shall" requirements with subsystem assignment.
STAGE 3
SSS
Unified system spec — dedup + conflict detection.
STAGE 4
Features
Capability groupings for product planning.
STAGE 5
PBIs
Backlog items with Given / When / Then ACs.

Every stage output is JSON-schema validated. Every requirement traces back to its customer ID. Nothing is silently dropped.

04 • Step-by-Step

Step 1 — Create venv & install

cd C:\Amp_demos\ELIRAN_AI\satcom_requirements_agent_harness
python -m venv .venv
.\.venv\Scripts\python.exe -m pip install -r requirements.txt

What this does:

  • Creates an isolated Python environment in .venv/ so the harness doesn't pollute your global Python.
  • Installs the only runtime dependency: jsonschema, used by the validator to enforce strict per-stage schemas.
  • No network calls beyond pip's package fetch — safe for air-gapped operation once the wheel is cached.
04 • Step-by-Step

Step 2 — Run the pipeline

.\.venv\Scripts\python.exe src\main.py --config config.json

The runner is src/main.py. Internally:

  1. Loads config.json.
  2. Picks MockLLMClient when mode: mock, or LocalLLMClient for production.
  3. Calls run_pipeline(config, llm) in src/pipeline.py.
  4. Writes 8 JSON files to outputs/.
  5. Prints score_report.json to stdout. Exits 0 on pass, 1 on fail.
Stage 1

Ingestion — deterministic by design

What it does

Scans every .md file under input/customer_requirements/ and extracts each requirement using the regex ^([A-Z]-REQ-\d{3}).

Why no LLM here?

Source-of-truth extraction must be reproducible. An LLM might paraphrase, drop, or invent IDs. Regex never does.

Output → stage1_ingested_requirements.json

{
  "id": "A-REQ-001",
  "text": "The system shall provide
           encrypted communication...",
  "source_document":
       "01_Customer_A_Defense_Requirements.md",
  "keywords": [],
  "domain": "satcom"
}
Stage 2

SRD — engineering normalization

What it does

For each customer requirement, produces a System Requirements Document entry: a single "the system shall…" statement, assigned to a subsystem, with a verification method.

Subsystem inference

Keyword-based router (infer_subsystem): "encrypt" → Security, "dashboard" → NMS, "bandwidth" → Ground Gateway, etc.

Output → stage2_srd.json

{
  "srd_id": "SRD-SEC-001",
  "text": "The system shall provide
           encrypted communication...",
  "source_ids": ["A-REQ-001"],
  "subsystem": "Security and Cryptography",
  "type": "functional",
  "verification": "test",
  "assumptions": [],
  "uncertainty": false
}
Stage 3 • The hero stage

SSS — unify, dedup, detect conflicts

Three jobs at once

  • Merge related SRDs from different customers into a single system-level requirement.
  • Detect conflicts when customers disagree (e.g. retention periods).
  • Maintain full traceability: source_srd_ids AND source_customer_ids are mandatory.

Conflict caught in this run

SSS-DATA-001
  "Retention periods differ by record class:
   operational logs require 7 years;
   billing usage records require 24 months."

  source_srd_ids:      [SRD-DATA-001, SRD-DATA-002]
  source_customer_ids: [A-REQ-013, B-REQ-009]

A real conflict between Defense (A) and Commercial (B) — surfaced automatically.

05 • Validation

Validation — what gets checked

Schema enforcement

  • Every artifact passes jsonschema.validate() against schemas.py.
  • Required fields, array minItems, types — strict.
  • additionalProperties: true only — never drop data, but never miss required fields.

Cross-reference checks

  • Every source_ids resolves to a real customer ID.
  • Every SSS has both source_srd_ids AND source_customer_ids.
  • Every Feature's linked_sss_ids resolve to real SSS items.
  • Every PBI's feature_id + linked_sss_ids resolve.
  • Every PBI acceptance criterion contains Given, When, Then.
  • Duplicate-text SSS items raise warnings.

08 • Compliance

Air-gapped, deterministic, auditable

No internet

The harness makes zero outbound connections. The only network operation is pip install — safe behind a corporate proxy or fully cached for air-gapped sites.

Deterministic

Mock mode produces byte-identical output every run, so CI can diff outputs/ against a golden reference.

Approved-model policy

config.json declares disallowed model origins (China) and families (Qwen, DeepSeek, GLM, Yi, Moonshot, Kimi, …) for the production LLM swap-in.

Strict spec constraints honored: no architecture changes, no removed validation, no simplified schemas, no pipeline replacement.

09 • Production Path

Swapping in a real local LLM

  1. Set "mode": "local" in config.json.
  2. Implement LocalLLMClient.generate_json() in src/llm_client.py — POST prompt + payload to your on-prem endpoint, parse JSON, return.
  3. Point llm.endpoint at your inference service: NVIDIA NIM, vLLM OpenAI-compatible, TensorRT-LLM, or a local Llama / Mistral wrapper.
  4. Keep schemas.py, validators.py, scoring.py, pipeline.py untouched.
  5. Re-run the same command. The runner picks up the new client automatically.

Because validation is downstream of generation, a misbehaving model is caught before scoring. Bad output → low score → CI gate fails.
